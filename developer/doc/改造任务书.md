

## 一、当前分支结论

`IELTS-WRITING-FEAT` 不应该推倒重写。它已经形成了一个相对正确的 Electron + Vue + Local API 架构。`package.json` 显示当前应用主入口是 `electron/main.js`，构建链路包含 `build:server`、`build:writing`、`start`、`release:build`，说明它已经不是单纯静态网页，而是 Electron 壳、本地 TypeScript server、Vue Practice Shell 的组合。 Vue 子应用使用 Vue 3、Vue Router 和 Vite，构建输出到 `dist/writing`。

Electron 侧的基础安全边界是正确的：主窗口启用了 preload，`contextIsolation: true`、`nodeIntegration: false`、`webSecurity: true`，这对学生端是必要的。 preload 暴露的能力也比较收敛，目前主要是打开 Practice 路由、打开阅读题、获取版本、获取 userData 路径、获取本地 API 地址。

本地 API 链路也已经成型。Electron 主进程初始化 `ServiceBundle`，启动 `LocalApiServer`，并把 API 地址通过 `app:getLocalApiInfo` 暴露给 renderer。 `LocalApiServer` 固定监听 `127.0.0.1`，优先端口 3000，被占用时 fallback 到随机端口。 Vue 侧 `client.js` 会通过 `window.electronAPI.getLocalApiInfo()` 解析本地 API baseUrl，并缓存到 sessionStorage/localStorage。

Practice API 是目前最重要的收敛入口。Fastify 注册了 `/api/practice/assets`、`/api/practice/assets/:activity/:assetId`、history、sessions、reading-suite、coach 等路线。 Vue 侧也已经有 `practice-client.js`，把 assets、sessions、readingSuite、history、coach、migration 分成了 API facade。 这条链路是正确方向，后续所有题库、NAS、历史、套题都应该继续走这条链路，不应该让 Vue 页面直接读取文件系统或执行外部 JS。

目前最大问题是：**“链路方向正确，但边界还不够干净。”** `PracticeLibraryPage.vue` 仍然是一个巨型页面，里面同时承载题库首页、浏览、练习历史、更多工具、设置、备份、题库配置、主题切换，并且保留大量 legacy DOM id、data attribute 和旧脚本加载逻辑。比如模板里同时包含 overview、browse、practice、more、vocab、settings 等大区块；脚本区还显式维护 `legacyMoreToolScripts`，动态加载一批旧 JS 文件。

`PracticeReadingPage.vue` 的状态和行为也已经很重。它直接承载计时器、答案同步、拖拽、笔记、高亮、生词气泡、复盘、AI Coach、套题模式、无尽模式、背题模式等逻辑。它通过 `practiceAssets.get('reading', normalizedAssetId)` 拉取单题 payload，这是正确的入口；但页面内部仍然直接维护大量 DOM bridge 和 legacy 兼容行为。

服务端同样需要收敛。`PracticeService` 现在同时处理 reading assets、writing topics、reading sessions、suite、history、archive import、legacy record import、AI coach。它还包含大量 legacy 记录归一化逻辑，例如从 `record.examId`、`record.assetId`、`metadata.examId`、`realData.examId` 等多处兜底推断旧记录。 这些兼容逻辑有价值，但不应该继续散落在核心服务里。

阅读题数据加载层目前是一个非常好的 NAS 改造切入点。`reading-assets.ts` 从 `assets/generated/reading-exams/manifest.js` 加载 manifest，然后读取具体题目 JS，并用 `parseReadingExamDataSource()` 解析受控注册格式；它不是直接 eval 整个 JS。 底层 parser 会找 `__READING_EXAM_DATA__.register(...)`，提取 key 和对象 payload，再进行 JSON.parse。 这说明你的 NAS 专版不必强行从 JS 改 JSON 文件；更合理的是保留 JS 文件形态，但把 JS 定义为“受控数据容器”，由服务端解析，不在 renderer 执行。

## 二、总改造原则

第一阶段目标是“核心链路收敛”，不改变产品功能，不接 NAS，不引入作者端，不改变客户交付形态。它只解决代码边界：让题库、阅读练习、套题、历史、设置、AI Coach、legacy adapter 各归其位。

第二阶段目标是“NAS 专版”。这个阶段必须在第一阶段完成后从稳定点新切分支，不能直接在当前巨型页面和巨型服务上叠 NAS，否则后面定位问题会非常痛苦。

核心原则如下：

第一，Vue 页面只消费 Practice API contract。Vue 不知道 manifest，不知道题目 JS 路径，不知道 NAS 路径，不知道 SQLite 路径，不直接执行外部 JS。

第二，服务端通过 Provider 抽象题库来源。当前内置题库是 `BuiltinReadingAssetProvider`；NAS 版以后新增 `NasSqliteReadingAssetProvider` 或 `NasJsReadingAssetProvider`。

第三，作者端可以继续产出 `.js`，但 `.js` 必须是受控注册数据格式。学生端不做 `<script src=NAS/xxx.js>`，而是由本地 API 读取文件并解析 payload。

第四，legacy 兼容只能存在于 Adapter 层，不能继续污染核心 PracticeService 和 Vue 页面。

第五，AI Coach 在核心链路收敛中应被隔离为插件能力。它可以保留，但不能和阅读基础练习、NAS 题库、历史记录强耦合。

### NAS-01：定义作者端发布产物

你当前 `IELTS-PDF2Test` 应升级为“作者端发布器”。客户不需要写 manifest，不需要写 SQL，不需要懂 JSON。

客户操作流程：


作者端必须做：

扫描 `source/*.js`；解析受控 register payload；校验 schema；计算 sha256；写入 `library.next.db`；校验通过后原子替换 `library.db`；写 `library.version.json`。

### NAS-02：保留 JS 题目源，但禁止 renderer 直接执行

这是本项目的关键决策。

你可以继续使用 `.js` 作为题目文件。但它必须是受控数据注册格式：

```js
__READING_EXAM_DATA__.register("p1-high-001", {
  "schemaVersion": "ReadingExamSourceV1",
  "examId": "p1-high-001",
  "meta": {},
  "passage": { "blocks": [] },
  "questionGroups": [],
  "answerKey": {},
  "questionOrder": []
});
```

学生端不使用 `<script src="NAS/xxx.js">`。学生端本地 API 读取 JS 文本，再用现有类似 `parseGeneratedRegisterSource()` 的方式提取 payload。该解析器当前已经存在，并且只提取受控注册调用中的 JSON payload。

如果做 SQLite 发布，学生端甚至不需要再解析每个 JS，只读作者端已经解析好的 `payload_json`。

### NAS-03：作者端 SQLite schema

推荐 `library.db` schema：

```sql
CREATE TABLE library_meta (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);

CREATE TABLE reading_assets (
  id TEXT PRIMARY KEY,
  exam_id TEXT NOT NULL,
  title TEXT NOT NULL,
  category TEXT,
  frequency TEXT,
  source_file TEXT NOT NULL,
  source_hash TEXT NOT NULL,
  source_size INTEGER NOT NULL,
  source_mtime INTEGER NOT NULL,
  payload_json TEXT NOT NULL,
  explanation_json TEXT,
  pdf_filename TEXT,
  status TEXT NOT NULL DEFAULT 'active',
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);

CREATE INDEX idx_reading_assets_status ON reading_assets(status);
CREATE INDEX idx_reading_assets_category ON reading_assets(category);
CREATE INDEX idx_reading_assets_hash ON reading_assets(source_hash);
```

`payload_json` 存 `ReadingPracticePayload` 的原始数据或 normalized 前数据。我的建议是作者端存“原始规范 payload”，学生端 provider 仍走一次 normalize，以保证兼容当前 `normalizeReadingDataset()` 的防御逻辑。当前学生端会 sanitize HTML、补 questionOrder、构建 interactionModel。

### NAS-04：学生端 NAS provider

新增：

```text
server/src/lib/practice/reading/NasSqliteReadingAssetProvider.ts
server/src/lib/practice/reading/NasLibraryConfigService.ts
server/src/routes/library.ts 或 server/src/routes/practice-library.ts
```

Provider 行为：

`listAssets()` 从本地缓存的 `library.db` 读取 `reading_assets`，返回 `PracticeAsset[]`。

`getAsset(assetId)` 读取 `payload_json`，走 normalize 后返回 payload。

`getStatus()` 返回 NAS 路径、当前版本、题目数量、最后同步时间、错误状态。

`refresh()` 检查 NAS 上 `library.version.json`，如版本变化，则把 `library.db` 复制到本地 userData cache，再切换本地 provider 数据源。

不要让 20 台学生机直接长期打开 NAS 上的 SQLite。学生端应复制到本地后只读。这样考试过程中即使 NAS 短暂抖动，本地缓存仍可用。

### NAS-05：Electron IPC 选择 NAS 目录

新增 preload API：

```js

getLibraryDirectory()
```

但不要把任意文件读写能力暴露给 renderer。renderer 只能让主进程打开目录选择器，主进程保存路径，Local API 读取状态。

## 六、关键取舍结论

第一，当前 `IELTS-WRITING-FEAT` 架构方向是对的，不要回到初代静态 Electron 版本。它已经有 Vue Practice Shell、本地 API、SQLite、Practice routes 和较好的 Electron 安全边界。

第二，当前分支确实需要二次重构，但目标不是“重新设计 UI”，而是“收敛核心链路”：Provider 化题库来源、Facade 化 PracticeService、Composable 化 Vue 业务状态、Adapter 化 legacy 兼容。

第三，NAS 专版不要和核心重构混在同一个分支。先把内置题库也抽象成 provider，再在新分支替换为 NAS provider。

第四，作者端可以继续生成 `.js`，但 `.js` 应被定义为“数据注册容器”。学生端不要在 renderer 直接执行 NAS JS；要么由本地 API 解析受控 JS，要么读作者端生成的 SQLite。这个策略和你现有 `parseGeneratedRegisterSource()` 方向一致，不是额外引入复杂度。

第五，`IELTS-PDF2Test` 后续应定位为“题库发布器”，而不仅是“PDF 转题工具”。客户只负责放题目 JS，作者端负责扫描、校验、发布，学生端只负责同步和读取。
